# ROADMAP v1 — Digital Billing Core (local slice)

**You are done with v1 when** a signed JioPay `0000` callback (running under `docker compose`, no cloud vendor) writes **exactly one** Order + Bill + Link + PENDING Broadcast in **one transaction**, **replaying the same `txnID` changes nothing**, `jbill.in/<identifier>` renders the bill in its template with working **PDF download + share**, and the scheduled worker drains the queue to **Mailhog** (FIFO; a FAILED row sets `attempts++` and retries later **without blocking** rows behind it) — with **no floats in the money path** and **no PII in logs**.

| ID | Step | Deliverable (one sitting) | Depends-on | Tier-1 | Verify locally |
|----|------|---------------------------|------------|--------|----------------|
| S-1 | Setup | `docker-compose.yml` brings up Postgres + Mailhog (local SMTP). | — | — | `docker compose up -d` → both containers healthy; `psql` connects; Mailhog UI opens at `:8025`. |
| S-2 | Setup | NestJS API app boots with a `GET /health` route. | — | — | `pnpm start:dev` → `curl :4000/health` returns `200`. |

| R-2 | 1 · Receive | `secureHash` guard: HMAC-SHA256, keys sorted alphabetically → values concatenated (no delimiter) → hex lowercase, keyed by merchant `SECRET_KEY`. **The sort-and-concatenate step MUST exclude the `secureHash` field itself** — it is part of the payload but not part of what is hashed. | M-1 | 2 · Persist | Money parser: rupee string (`"1.00"`) → `BigInt` paise via string/decimal parse — **never** `parseFloat*100`. | L-1 | 3 · Link | Collision-safe short-code identifier generator (URL-safe, e.g. nanoid/base62), pure util. *(Pulled early: the Link is minted **inside** the Step-2 transaction per D-6, so P-1 depends on it.)* | S-2 | — | `pnpm test link-id` → N generations are unique and URL-safe. |
| P-1 | 2 · Persist | Persist-on-success as a **single Prisma `upsert` on `txnId`**: Bill, Link, and the PENDING Broadcast are created via **nested writes inside that upsert's `create` branch**, and the `update` branch is **empty (`update: {}`)** for the Order and all three nested relations — so a redelivery **no-ops (touches nothing)** instead of erroring. Do **NOT** upsert the Order and then call separate `.create()`s for Bill/Link/Broadcast: on replay those hit the unique constraint, throw, roll back the transaction, and return `500`, which an at-least-once JioPay retries forever. **This nested-write-in-the-`create`-branch is the single most important implementation detail in this task.** Non-success → `Order(status=NON_SUCCESS)` only (no Bill/Link/Broadcast). | S-4, S-5, R-2, M-1, L-1 | **Tier-1 (money + PII)** | Signed `0000` callback → **exactly 1** Order/Bill/Link/Broadcast(PENDING); **replay the same `txnId` → HTTP `200` (not `500`) and all four row counts unchanged**; non-success callback → Order only. Confirm with `SELECT count(*)` / Studio. |
| L-2 | 3 · Link | Resolver returns a **whitelisted read-model / DTO** built with a Prisma `select` that lists **only** the fields the bill view needs: bill line items, amount, template, `identifier`, merchant **display name**. **Exclude** `customerMobile_pii`, `customerEmail_pii`, `rawCallback`, `secureHash`, and internal IDs. **Whitelist at the resolver — do not rely on the view/props layer to withhold fields** (`jbill.in/<code>` is unauthenticated and Next.js serializes props into client HTML). Unknown identifier → `404`. | P-1 | **Tier-1 (public unauth surface; PII leak)** | Resolve an identifier from P-1 → payload contains only the whitelisted fields; **assert `customerMobile_pii`/`customerEmail_pii`/`rawCallback`/`secureHash`/internal IDs are absent**; unknown code → `404`. |

| V-2 | 4 · View | Public page `jbill.in/<identifier>` (Next.js route) renders the bill via V-1 + L-2. | V-1, L-2 | **Tier-1 (renders customer PII to public page)** | `open :3000/<identifier>` → bill renders in its template; only intended fields shown (no secret/hash/raw-callback leakage); unknown identifier → `404`. |
| V-3 | 4 · View | PDF download of the rendered bill. | V-2 | — | Click **Download PDF** on the link-view page → a PDF file downloads containing the bill. |
| V-4 | 4 · View | Share action (copy/share the `jbill.in` link). | V-2 | — | Click **Share** → share sheet opens / link is copied. |
| B-1 | 4 · Broadcast | Stubbed send adapter behind a single port/seam that **dispatches by `Order.channel`**: `email` → send via local SMTP to **Mailhog** (no real vendor); `sms` → **log-only stub** (masked log line, no external call). Both channels return a stub "sent" result; recipient masked in logs for both. | S-1 | **Tier-1 (recipient is PII)** | Call the adapter with an `email`-channel row → email appears in Mailhog UI (`:8025`), masked log line; call it with an `sms`-channel row → masked log line only (no external call) and the stub reports it sent. |
| B-2 | 4 · Broadcast | Scheduled NestJS `@Cron` drainer: pick oldest eligible row (**FIFO by `createdAt`**), send via B-1; success → `SENT` + `sentAt`; failure → `FAILED` + `attempts++` + `error`, retried on a later pass up to **max-attempts (config knob — default 5; see D-7)**; a FAILED row does **not** block rows behind it. | B-1, P-1 | **Tier-1 (recipient is PII)** | Seed PENDING rows → tick → oldest → `SENT` (email in Mailhog); force a failure → that row `FAILED`, `attempts=1`, later rows still `SENT`; next tick retries it; exceeding max-attempts stops retrying. |
